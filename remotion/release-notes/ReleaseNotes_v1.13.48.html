[COMMONS-2349] IClientTransactionExtension and IClientTransactionListener.FilterQueryResult notifications are now executed in sub-transactions as well as root transactions

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed


Previously, only root transactions would raise the FilterQueryResult notification. Since all transactions in a hierarchy share the same extensions, this didn't really matter a lot - filtering was performed in the root transaction anyway.

It was highly asymmetric when compared with all other notifications, though. This is the reason why it was changed. Extensions should therefore decide for themselves how to react on FilterQueryResult notifications in subtransactions by checking the clientTransaction parameter. SecurityClientTransactionExtension, for example, ignores FilterQueryResult in the sub-transaction; it performs all filtering in the root transaction.

[COMMONS-2336] DomainObjectCollection.AsReadOnly and SetIsReadOnly have been removed

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed


SetIsReadOnly has been removed because it's not desirable (and potentially dangerous) to be able to make read-only collections writable again. Create read-only collections via Clone or DomainObjectCollectionFactory.CreateReadOnlyCollection.

AsReadOnly has been replaced by AsReadOnlyCollection, which returns a ReadOnlyCollection<T> as an adapter of the original collection. For an adapter with a bigger feature set, use ReadOnlyDomainObjectCollectionAdapter<T>.

[COMMONS-2288] Queries performed in sub-transactions now immediately load the result objects into the sub-transaction

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed


Previously, queries performed in sub-transactions would only load the result objects into the root transaction; the sub-transaction would enlist but not load the objects.

This is changed in order to achieve better symmetry between root transactions and sub-transactions.

[COMMONS-2253] DomainObjectCollection.AssociatedEndPoint has been replaced by AssociatedEndPointID and IsAssociatedWith

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed

The reason for this change is that relation end points should only be accessible by clients deriving from ClientTransaction; ordinary domain users should not have access to the internal management data structures.

[COMMONS-2345] Public ExpressionTreeVisitor.VisitList and VisitAndConvert methods

Component/s: Data.Linq
Issue Type: New Feature
Resolution: Fixed
Status: Closed


public static ReadOnlyCollection<T> VisitList<T>(ReadOnlyCollection<T> list, Func<T, T> elementVisitor);
public static T VisitAndConvert<T> (T expression, string caller) where T : Expression;
public static ReadOnlyCollection<T> VisitAndConvert<T>(ReadOnlyCollection<T> list, string caller) where T : Expression;

VisitList applies the elementVisitor function to each member of the list, returning the same list if each visitor call returned the original item. Otherwise, it returns a new list with the new items replacing the corresponding original ones.

VisitAndConvert for a single expression calls VisitExpression on the expression, then tries to cast it back to the type T. If the cast doesn't succeed (because the visitor's result is of a new type or null), an exception is thrown.

VisitAndConvert for lists combines VisitList with the single-expression VisitAndConvert method.

[COMMONS-2240] ClientTransactionand DomainObjectTransactionContext.Execute methods

Component/s: Data
Issue Type: New Feature
Resolution: Fixed
Status: Closed


Often, it's necessary to execute a statement or calculate an expression in the context of a certain ClientTransaction. To do so, one has to spawn a new scope:

Order order;
using (transaction.EnterNonDiscardingScope())
{
  order = Order.NewObject();
}

Using the ClientTransactionExtensions.Execute method, this becomes a one-liner:

var order = transaction.Execute (() => Order.NewObject());

Similarly, DomainObjectTransactionContext.Execute can be used to execute a piece of code in the context of the transaction associated with a DomainObject:

// set the OrderTicket in the binding transaction of order, if any
// otherwise, use the current transaction
order.DefaultTransactionContext.Execute ((obj, tx) => order.OrderTicket = OrderTicket.NewObject());

// commit the binding transaction of order if any
// otherwise, commit the current transaction
order.DefaultTransactionContext.Execute ((obj, tx) => tx.Commit());

[COMMONS-2313] New Sql Backend

Component/s: Data.Linq
Issue Type: New Feature
Resolution: Unresolved
Status: Open
FixVersion: re-linq vNext


The new SQL backend should both be a good example of how to approach SQL generation using the new re-linq front-end (QueryModel, usw.) and provide a sound, extensible architecture backing re-store's LINQ provider.

It will sport SQL generation in three phases:

  • The first phase will take a query model and translate it into a SqlStatement model. SqlStatement is modeled to express the syntactical possibilities and constraints of actual SQL statements.
  • The second phase will allow an O/R mapper to resolve all elements of the SqlStatement into SQL-specific items such as tables or columns with names and aliases.
  • The third phase will then take the resolved SqlStatement and produce actual SQL text (and parameters).

LINQ providers will be able to hook into each of the three steps for customization, optimization, and advanced query transformations.
