[COMMONS-1538] BREAK: For generic, auto-closed mixins, require more explicit specification what to do with the type arguments

Component/s: Mixins
Issue Type: Improvement
Resolution: Fixed
Status: Closed


Currently, a mixin such as the following is instantiated using heuristics:

class GenericMixin<T1, T2> : Mixin<T1, T2>
where T1 : X
where T2 : Y

The algorithm to find an instantiation for a type parameter T is as follows:

  • From the constraints of T, form a candidate. If the constraints cannot be unified into a single candidate (e.g. if there is more than one interface, or there are a class and an interface that are not assignment-compatible), an exception is thrown.
  • If the parameter is not bound to the TBase parameter of Mixin<TThis, TBase>, check whether the candidate is assignment-compatible with the target type of the mixin. If yes, the target type becomes the candidate.
  • If the candidate is ValueType (because there was only a "struct" constraint), use typeof (int).
  • If no candidate was found, use typeof (object).
  • Otherwise, use the candidate.

This is very implicit, and, above all, it changes from:

class GenericMixin<T> : Mixin<T>

to

class GenericMixin<T> : Mixin<T, T>

(In the first case, T becomes the target type; in the second, it becomes typeof (object).)

Therefore, use a more explicit approach via custom attributes:

class GenericMixin<[BindToTargetType] T1,
                   [BindToConstraints] T2> : Mixin<T1, T2>
  • BindToTargetType binds to the target type (throwing an exception if there are non-matching constraints).
  • BindToConstraints infers the type to bind to from the constraints (without regarding whether it's used as TThis or TBase).
