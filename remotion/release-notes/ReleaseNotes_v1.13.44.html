[COMMONS-2219] DomainObjectCollection.Commit method now takes an IEnumerable<DomainObject> instead of a DomainObjectCollection

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed


This was done because the original DomainObjectCollection should not access the collection representing the relation end point (possibly causing RelationRead events, etc.) during its Commit method.

[COMMONS-2218] IClientTransactionListener.RelationRead and IClientTransactionExtension.RelationRead notification methods for collection-valued relations have changed

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed


Those methods now take a ReadOnlyDomainObjectCollectionAdapter<DomainObject> instead of a ReadOnlyCollection<DomainObject>. The adapter provides more information about the collection, most notably the IsDataAvailable property. Implementors should check this property before accessing the collection contents if they want to avoid reloading collections of unloaded end points.

[COMMONS-2207] RepositoryAccessor has been renamed to LifetimeService and moved to the new namespace Remotion.Data.DomainObjects.DomainImplementation

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed


All advanced services for domain writers are moved to the new DomainImplementation namespace. The new name seems to be a better fit, given that there is no concept of "Repository" in re-store.

[COMMONS-2190] Changes to IClientTransactionExtension.ObjectLoading and IClientTransactionListener.ObjectLoading

Component/s: Data
Issue Type: Breaking Change
Resolution: Fixed
Status: Closed


Previously, the ObjectLoading method took a single ObjectID indicating what object was about to be loaded. This is inconsistent with the ObjectsLoaded method, which takes an array of DomainObjects. When several objects were loaded, ObjectsLoaded would be called only one, but ObjectLoading would be called multiple times.

ObjectLoading has now been replaced with ObjectsLoading which works similar to ObjectsLoaded.

[COMMONS-2104] Unload events

Component/s: Data
Issue Type: New Feature
Resolution: Fixed
Status: Closed


The following events will be raised when one or more objects' data is unloaded via DomainObjectUnloader.UnloadData or UnloadCollectionEndPointAndData (in this order):

  • IClientTransactionListener.ObjectsUnloading/IClientTransactionExtension.ObjectsUnloading
  • DomainObject.OnUnloading (once for each object whose data is unloaded; at this point of time, the current transaction is guaranteed to be the transaction from which the object is unloaded)
  • Data and end points are unregistered
  • DomainObject.OnUnloaded (once for each object whose data is unloaded; at this point of time, the current transaction is guaranteed to be the transaction from which the object is unloaded)
  • IClientTransactionListener.ObjectsUnloaded/IClientTransactionExtension.ObjectsUnloaded

ClientTransactionListener.ObjectsUnloading, IClientTransactionExtension.ObjectsUnloading, and DomainObject.OnUnloading can throw an exception in order to cancel the operation.

Calls to DomainObjectUnloader.UnloadCollectionEndPoint will not raise any notifications at this point of time.

[COMMONS-2090] Maybe type

Component/s: Core
Issue Type: New Feature
Resolution: Fixed
Status: Closed


The Maybe type stores a value that may be null and provides functionality to process and act on this value without having to manually perform null checks:

public Maybe<int> Square(Maybe<int> value)
{
  return value.Select (i => i * i);
}

Use this for chained expressions, where each of the pieces could be null:

var x = Maybe.ForValue (Control.DataSource).Select (ds => ds.BusinessObjectClass).Select (c => c.Identifier).ValueOrDefault("");

Use the Where method to perform checks on the value stored by a Maybe instance, use the Do and Otherwise methods to perform actions based on whether or not the Maybe instance has a value.

[COMMONS-1792] ClientTransaction.HasRelationChanged: change state of collection-valued relations of a domainobject should be cached

Component/s: Data
Issue Type: Performance
Resolution: Fixed
Status: Closed


Note that there may be code that directly accesses IEndPoint.HasChanged or CollectionEndPoint.HasChanged. That code should also profit from HasChanged caching; therefore we should probably implement caching at the end point level, not at the ClientTransaction level.

A simple implementation would be to store a changed flag in the end point and have EndPointDelegatingCollectionData invalidate the flag on every change.
